## Current Issues

### rmcp 1.8.0 API Type Mismatches (10+ errors)

**Compilation Errors Found:**
1. `McpError` import path incorrect — need to find correct import
2. `Tool` struct field names don't match expected (`inputSchema` not found)
3. `Tool` struct is non-exhaustive — cannot use struct expression syntax
4. `ListToolsResult` field name incorrect (`nextCursor` not found)
5. `CallToolResult::success()` and `error()` take `Vec<Content>` not `TextContent`
6. `TextContent` struct doesn't have `text` field — uses `Annotated<RawTextContent>` pattern
7. `TextContent` creation syntax incorrect

**Root Cause:**
- rmcp uses macro-generated types with different field names and patterns
- Tool definitions are generated by `#[tool]` macro in rmcp-macros
- Response content uses `Content` enum instead of direct `TextContent`
- Need to understand rmcp's actual type definitions and creation patterns

## Key Findings

### rmcp 1.8.0 Architecture
- Uses trait-based handler pattern with `ServerHandler` trait
- Tool definitions are generated by `#[tool]` macro in rmcp-macros crate
- Response content uses `Content` enum with variants for different content types
- `ListToolsResult` is generated by `paginated_result!()` macro
- `Tool` struct is non-exhaustive and requires builder pattern or special constructor

### Correct Approach
1. Implement `ServerHandler` trait with required methods
2. Use rmcp's `model` types for requests/responses
3. Define tools with proper schemas
4. Route tool calls to MemoryOrchestrator
5. Return responses in rmcp's `Content` format
6. Use `serve_directly()` to run the server with stdio transport
